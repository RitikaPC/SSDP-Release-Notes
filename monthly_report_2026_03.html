Release Summary

➕ What we added:

 Targeted Maintenance Release: This specific release cycle focused exclusively on resolving a critical display bug within the vehicle interface, so no new functional features were introduced,

  • VDP_STORE_2 (VDS2) Foundational Rollout: A massive data-layer deployment introducing full support for core vehicle domains in the new architecture. This includes the integration of State, Telemetry, Position, Stolen, Alarm, Alert, and Trip features.

  • Universal Charging Sessions: Enabled centralized Charging Session tracking for all architectures directly within VDS2.

  • B2B Ecosystem Support: Implemented comprehensive B2B data support within VDS2 to ensure enterprise fleet data is captured and loadable by MPH APIs.

  • Real-time Push Notifications (DOCG): Configured DOCG to trigger a push notification every time one is requested by the OTA Redbend (RB) Server within the operation topic, improving real-time user feedback during campaigns.

  • Enhanced Telemetry Data: Added the Electric (ELEC) battery temperature block and mapping for Carbide architectures, and integrated Stolen vehicle data directly into the SSDP Telemetry collection.

  • Targeted Maintenance Release: No net-new user-facing features were introduced in this cycle. The focus was strictly on restoring core data layer functionalities and preparing specific environment configurations.

  • This release period (Week 13, 2026) focused primarily on infrastructure optimization and resolving urgent production workflow blockers rather than introducing new end-user features.

🔧 What we changed:

  DOCG Version Bump: Upgraded the DOCG component from version 26.1.2.17 to 26.1.2.18 (baseline 1.8.3) to deploy the necessary UI fix. All other ecosystem enablers (APIM, EAH, VDR, PATRIC, etc.) remain unchanged on their current stable versions.

  • Data Storage & Retention Strategy: Completely overhauled document identification in VDS2 (replacing system _id with a functional ID) and introduced a dynamic TTL (Time-To-Live) / Expiration strategy defined per FleetID and feature.

  • Traffic Filtering: Implemented strict source filters in VDS2 to discard non-CARCOM source data flows, reducing system noise and preventing the storage of useless telemetry data.

  • Performance Testing Prep (PATRIC): Refactored the Integration (INT) environment with database exports and a bridge of production sampling to prepare for heavy B2C performance benchmarking.

  • Consumer Repartitioning: Optimized Kafka/Elastic consumer repartitioning and resolved Elastic agent startup errors for better data ingestion stability.

  • Environment Configuration (VDP_PROC): Prepared and isolated the production configuration specifically for F2X deployments, ensuring tailored processing rules for this architecture.

  • Kafka Optimization: Refactored the Kafka consumer commit logic within the VDP_PROC (PROC2) stream to improve message processing reliability and prevent data consumption lags.

  • We successfully upgraded the DOCG component to the 1.9.0 baseline, specifically deploying version DOCG-26.1.3.14 on March 27, 2026.

  • We implemented a critical infrastructure improvement by increasing the DOCG Database Maximum-pool-size to 30, which will enhance connection management and overall system performance.

  • In parallel, we rolled out an urgent Hotfix version (DOCG-26.1.2.20-1.8.4) directly to production to address immediate service activation blockers.

🚮 What we Deprecated/ Removed:

  • VDS1 Charging Sessions: With the successful transition to VDP_STORE_2, charging session storage in the legacy VDS1 environment has been officially disabled.

🔨 What we fixed:

IVI Release Notes Visibility (CSERV-1997): Resolved a critical issue in the ICON Preprod environment where release notes and descriptions configured within the DOCG Backoffice were failing to render on the In-Vehicle Infotainment (IVI) screens. This fix ensures that end-users will correctly see the necessary descriptions and instructions for their OTA updates directly in their vehicles.

  • Data Privacy Compliance: Fixed a critical issue in VDP_STORE_2 where privacy values in the CO and COS objects were not updating to reflect the customer's active choices.

  • B2B & B2C Sync Issues: Resolved multiple database update failures where B2B data was freezing, and B2C charging sessions were failing to store properly.

  • Monitor Reliability (PATRIC): Fixed a bug where Monitor Start/Stop triggers were failing to execute for both B2B and B2C flows.

  • Campaign & Inventory Accuracy: Patched the DOCG Post-Op flow to prevent the system from receiving and sending multiple redundant inventories during a single operation.

  • Telemetry Visibility: Resolved API errors preventing the retrieval of telemetries/latest (specifically missing stolen info) and filtering issues within the Trips API.

  • Data Visibility & Fleet Filtering (HOTFIX): Resolved a critical production issue in VDP_STORE_2 where the Fleet filter was exclusively applying to the "State" feature. This hotfix ensures fleet-based data segmentation is correctly applied across all telemetry and vehicle data features.

  • Energy Data Integrity (HOTFIX): Patched a bug causing failures in "energy merging" within VDP_STORE_2, ensuring that electric and fuel energy data streams are correctly combined and stored without loss or corruption.

  • Resolved an urgent production issue where the DOTE Scomo was not being added to campaigns.

  • Corrected a critical flaw in the Service Activation (OTA) process where "cancel" or "TimeOut" statuses were being ignored, which previously left workflows stuck in an open state.

  • Patched the ADA and DM Service Activation pipeline to ensure data is successfully sent to Account Management (AM) and that correlation IDs are properly assigned for DM workflows.

  • Fixed the OTA Service Activation feedback loop so that AM Feedback is accurately sent even when a Redbend (RB) cancellation or failure status occurs
