Release Summary

➕ What we added:

  • New Architecture Configurations (VDP_PROC): Expanded our processing capabilities by adding specific configurations to support new vehicle architectures (rtbmv1-nea10 and rtbmv1-nea11).

  • Dedicated F2X Environment: Prepared and isolated a production configuration specifically tailored for F2X deployments within the VDP_PROC stream.

  • Message Traceability (DOCG): Configured the system to add a Kafka Key for every message published by the JMSAdapter within the campaign flow, improving message tracking and partitioning. (Note: This was part of a rolled-back hotfix).

🔧 What we changed:

  • Kafka Consumer Optimization: Refactored the Kafka consumer commit logic within the VDP_PROC stream to improve message processing reliability.

  • Version Advancements: Advanced the DOCG component to the 1.9.1 baseline and rolled out updates to VDP_PROC (1.5.5 and 2.33.1).

🚮 What we Deprecated/ Removed:

  • Deployment Rollback: The DOCG hotfix release (DOCG-26.1.3.17 - 1.9.1) was rolled back from production.

  • No explicit feature deprecations were executed in this release cycle.

🔨 What we fixed:

  • System Resource Exhaustion (DOCG): Addressed critical infrastructure issues where DOCG PODs were consuming excessive memory (affecting almost all applications) and fixed the malfunctioning Production DB Max Pool update.

  • Alerts Data Integrity (VDP_PROC): Resolved a mapping issue for J4U Alerts across both PROC1 and PROC2 streams in the production environment.
